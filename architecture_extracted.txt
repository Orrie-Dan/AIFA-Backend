1. System Overview
AIFA is a financial intelligence platform with an AI interface — not an AI chatbot with financial data attached. The distinction matters architecturally: all numerical reasoning, calculations, and financial logic lives in the Financial Engine. The LLM is a presentation layer that translates engine output into natural language.
Core Design Principles
Principle
Implication
Numbers never come from the LLM
All calculations are done in the backend; the LLM only formats results
Privacy by default
No raw transaction data leaves your infrastructure without explicit consent
Rwanda-first
Algorithms account for mobile money dominance, informal income, and RWF denomination
Fail safe
Any LLM output touching a number must be validated before display
Sparse data tolerance
ML models are only introduced when you have enough per-user history to outperform simple baselines

3. Data Privacy & Compliance
This is the most under-specified area in typical fintech AI designs. Getting it wrong destroys user trust.
3.1 Data Classification
TIER 1 — NEVER LEAVES YOUR SERVERS
───────────────────────────────────
- Full transaction records
- Account balances
- Bank account details
- Mobile money numbers (MTN MoMo, Airtel)
- Names linked to transactions
- Raw category labels with merchant names
TIER 2 — AGGREGATED, CAN BE SENT TO LLM (anonymized)
──────────────────────────────────────────────────────
- Monthly spending totals per category
- Savings rate (%)
- Budget adherence score
- Financial health score (numeric, no detail)
- Goal progress (% complete, not amounts unless user consents)
TIER 3 — USER-CONTROLLED (explicit consent required)
──────────────────────────────────────────────────────
- Exact amounts for specific questions
- Full category breakdowns
- Income figures
3.2 What the LLM Actually Receives
The LLM never receives a raw transaction list. It receives a structured, pre-computed summary:
{
  "context_type": "affordability_query",
  "period": "2025-01",
  "financial_snapshot": {
    "monthly_net_income_band": "800k-1M RWF",
    "savings_rate_pct": 28,
    "top_spending_categories": [
      { "category": "Food", "pct_of_income": 22 },
      { "category": "Transport", "pct_of_income": 8 }
    ],
    "months_to_goal": 4,
    "engine_result": { "affordable": true, "confidence": "high" }
  },
  "user_question": "Can I afford a 1,200,000 RWF laptop by December?"
}
No transaction IDs. No merchant names. No exact account balance. Amounts are banded unless the user explicitly consented in Settings to share exact figures.
3.3 Two-Mode LLM Strategy
User Onboarding

     ▼

Both modes are honest about their trade-offs. Do not hide Mode B behind settings — make it a first-run choice.
3.4 Data Retention for LLM Calls
OpenAI API calls: set user field to a hashed, non-reversible user token (not actual user ID)
Enable OpenAI's zero-data-retention option if your tier allows it
Log all LLM prompts/responses server-side for audit, encrypted at rest
Purge LLM prompt logs after 90 days
4. Financial Intelligence Engine
All calculations here. No LLM involvement.
4.1 Affordability Analysis
INPUTS
──────
current_monthly_net_income: RWF
monthly_fixed_expenses: RWF       (rent, subscriptions, loans)
monthly_variable_avg_3mo: RWF     (3-month rolling average of variable spend)
current_liquid_savings: RWF
target_item_price: RWF
target_month: Date
emergency_fund_target: RWF        (default: 3× monthly expenses)
ALGORITHM
─────────
months_available = (target_month - today).months
monthly_disposable = net_income - fixed_expenses - variable_avg_3mo
emergency_fund_current = liquid_savings - (monthly_fixed_expenses × 3)
  → if emergency_fund_current < 0, reduce disposable by shortfall/12
projected_savings = liquid_savings + (monthly_disposable × months_available)
affordable = projected_savings >= target_item_price
buffer_after = projected_savings - target_item_price
buffer_ratio = buffer_after / (monthly_fixed_expenses × 3)
confidence:
  "high"   if months_available >= 3 AND income_variance_3mo < 15%
  "medium" if months_available >= 2 OR income_variance_3mo 15–30%
  "low"    if months_available < 2 OR income_variance_3mo > 30%
OUTPUT
──────
{
  "affordable": bool,
  "months_needed_minimum": int,
  "projected_savings_at_target_date": RWF,
  "buffer_after_purchase": RWF,
  "emergency_fund_impact": "none" | "partial" | "depleted",
  "confidence": "high" | "medium" | "low",
  "warnings": []   ← e.g., "Income has been irregular last 3 months"
}
4.2 Spending Analysis
Do not use ML here until Phase 4. Use statistical baselines:
For each category C:
  current_month_spend = sum(transactions[C][this_month])
  baseline = rolling_average(transactions[C], last_3_months)
  std_dev = std(transactions[C], last_3_months)
  z_score = (current_month_spend - baseline) / std_dev
    → |z| > 1.5: "Notable increase/decrease"
    → |z| > 2.5: "Unusual spike — flag for user"
  mom_change_pct = (current - baseline) / baseline × 100
This beats ML for users with < 6 months of history (which is most of your early users).
4.3 Expense Forecasting (Phase 4 only)
Introduce ML only when a user has ≥ 6 months of data. Before that, use:
next_month_forecast[C] = 
  weighted_average(last_3_months[C], weights=[0.5, 0.3, 0.2])
  + seasonal_adjustment[C][month_of_year]
Build seasonal_adjustment from aggregate anonymized data across users (same category, same calendar month).
When introducing ML:
Model: LightGBM (not XGBoost — faster to iterate, better on sparse data)
Features:
  - day_of_week, day_of_month, month
  - rolling_avg_7d, rolling_avg_30d
  - is_salary_week (boolean)
  - category_encoded
  - user_tenure_months (proxy for data reliability)
Target: next_30d_spend_by_category
Validation: hold out last month, measure MAE per category
Ship only if MAE < baseline (weighted rolling average)
4.4 Anomaly Detection
Rule-based (Phase 1–3):
  - Transaction > 3× average for that merchant → flag
  - Same amount, same merchant, within 24h → possible duplicate → flag
  - Transaction at unusual hour (midnight–5am) → soft flag
Statistical (Phase 3+):
  - IQR method per merchant/category
  - Any transaction outside [Q1 - 1.5×IQR, Q3 + 1.5×IQR] → flag
4.5 Transaction Categorization
Priority order:
  1. User-confirmed category (stored, never overridden)
  2. Merchant lookup table (build your own: MTN MoMo, Equity Bank, 
     Simba supermarkets, Nakumatt, etc.)
  3. Keyword rules (e.g., "KIGALI BUS" → Transport)
  4. LLM classification (fallback, cached per merchant name)
     → Send only merchant name, not amount or date
     → Cache result so same merchant never hits LLM twice
  5. "Uncategorized" (user prompted to categorize)
5. Rwanda-Specific Financial Context
This shapes algorithm design significantly. Ignoring it produces a product that feels foreign.
5.1 Income Patterns
Pattern
Implication for AIFA
Many users have informal/variable income
Income field should allow "estimated monthly range" not just fixed salary
Salary often weekly or bi-weekly for informal sector
Cashflow analysis must detect pay cycles automatically
Multiple income sources common
Wallet model must support income tagging per transaction
Mobile money is primary banking for many
MTN MoMo / Airtel Money transactions must be first-class, not an afterthought
Income detection algorithm:
Every 30 days, run:
  large_inflows = transactions WHERE amount > (avg_transaction × 5) 
                              AND type = "credit"
  cluster by day_of_month (tolerance ±3 days)
  if cluster has ≥ 2 occurrences → suggest as income source
  → prompt user to confirm: "Is this your salary from X?"
5.2 Mobile Money Integration
Transaction Sources to Support:
  - MTN MoMo (primary — largest user base in Rwanda)
  - Airtel Money
  - Equity Bank mobile
  - Bank of Kigali mobile
  - Manual entry (always required as fallback)
MoMo SMS Parsing (Phase 2):
  Pattern: "You have received RWF [AMOUNT] from [NAME] ([PHONE]).
            Your new balance is RWF [BALANCE]."
  
  → Parse with regex, confirm with user before importing
  → Strip sender name before any LLM processing (privacy)
  → Store hashed phone numbers for deduplication only
5.3 Currency and Amounts
All storage in RWF (integer — no decimals, RWF has no fils)
All display in RWF with thousand separators: 1,200,000 RWF
Budget thresholds calibrated for Kigali cost of living: 
Food: 100k–400k RWF/month typical range
Transport: 20k–150k RWF/month
Rent: 100k–500k RWF/month (varies hugely)
5.4 Financial Health Score — Rwanda Calibration
(Full design in §6)
Emergency fund target = 3× monthly expenses is a Western standard. For users with mobile money as primary account and irregular income, 2× is a more realistic initial target with 3× as the aspirational goal. The score should reflect this.

6.1 Score Components
TOTAL SCORE: 0–100
Component weights:

│ Component               │ Weight │ Why this weight                 │
├─────────────────────────┼────────┼─────────────────────────────────┤
│ Savings Rate            │  30%   │ Primary driver of financial      │
│                         │        │ security                        │
├─────────────────────────┼────────┼─────────────────────────────────┤
│ Emergency Fund Coverage │  25%   │ Critical buffer, esp. for       │
│                         │        │ irregular income                │
├─────────────────────────┼────────┼─────────────────────────────────┤
│ Budget Adherence        │  20%   │ Behavioural consistency         │
├─────────────────────────┼────────┼─────────────────────────────────┤
│ Income Stability        │  15%   │ Rwanda-specific: income         │
│                         │        │ variance matters more here      │
├─────────────────────────┼────────┼─────────────────────────────────┤

│                         │        │ common, but mobile loans growing│
└─────────────────────────┴────────┴─────────────────────────────────┘
6.2 Component Scoring Functions
SAVINGS RATE (SR)
  savings_rate = monthly_savings / monthly_gross_income
  Score:
    SR ≥ 30%  → 100
    SR 20–29% → 80
    SR 10–19% → 60
    SR 5–9%   → 40
    SR 1–4%   → 20
    SR < 1%   → 0
  Note: Target thresholds set 5–10% lower than typical Western benchmarks
  to reflect higher cost-of-living pressure relative to income in Rwanda.
EMERGENCY FUND COVERAGE (EF)
  ef_months = liquid_savings / monthly_expenses
  Score:
    ef ≥ 3.0  → 100
    ef 2.0–2.9 → 80   ← Rwanda minimum target
    ef 1.0–1.9 → 55
    ef 0.5–0.9 → 30
    ef < 0.5  → 0
BUDGET ADHERENCE (BA)
  For each budget category C with a set limit:
    adherence[C] = min(1.0, budget_limit[C] / actual_spend[C])
  BA = weighted_average(adherence, weights=budget_limits)
  Score:
    BA ≥ 95%  → 100
    BA 85–94% → 80
    BA 70–84% → 60
    BA 55–69% → 40
    BA < 55%  → 20
INCOME STABILITY (IS)
  cv = std(last_6_months_income) / mean(last_6_months_income)
  (coefficient of variation)
  Score:
    cv < 0.10 → 100  (very stable)
    cv 0.10–0.19 → 80
    cv 0.20–0.29 → 60
    cv 0.30–0.49 → 40
    cv ≥ 0.50   → 20  (highly variable — common in informal sector)
  Note: Low score here does NOT mean user is irresponsible.
  AI explanation must acknowledge income variability, not blame.
DEBT SERVICING RATIO (DS)
  dsr = total_monthly_debt_payments / monthly_gross_income
  Score:
    DSR < 10%  → 100
    DSR 10–19% → 80
    DSR 20–29% → 60
    DSR 30–39% → 40
    DSR ≥ 40%  → 0
  Includes: formal loans, mobile loan apps (Tala, Branch, etc.)
6.3 Score Handling Edge Cases
Situation
Handling
< 1 month of data
No score. Show "Building your profile..."
No budgets set
Budget adherence excluded from score, other components reweighted
No debt data
Debt component excluded, reweighted
Income = 0 this month
Flag as data issue before scoring

6.4 Score Display Guidelines
Show score as a number (0–100) AND a band label
Band labels must be neutral, not judgmental: 
80–100: "Strong"
60–79: "Building"
40–59: "Developing"
0–39: "Needs Attention"
Always explain the top driver and top improvement action
Never show score without a one-line explanation of what changed
7. LLM Integration Strategy
7.1 Context Management
The core problem: Each LLM call is stateless. The LLM remembers nothing between sessions. Sending the entire transaction history on every call is expensive, slow, and a privacy risk.
CONTEXT ARCHITECTURE
─────────────────────
Persistent Context (stored in Redis, refreshed daily):
  - Financial snapshot (aggregated, see §3.2)
  - Active goals summary
  - Budget summary (categories + adherence %)
  - Health score + trend
  - User preferences (e.g., "prefers concise answers")
Session Context (in-memory for current conversation):
  - Last 10 conversation turns (sliding window)
  - Any entities mentioned this session ("the laptop", "my December goal")
Per-Request Context (injected per tool call result):
  - Raw engine output for this specific query
  - Relevant supporting data (max 3 months of category data)
TOTAL CONTEXT BUDGET: ~2,000 tokens per call (well within limits)
7.2 System Prompt (Canonical)
You are AIFA, a personal financial advisor for users in Rwanda.
STRICT RULES:
1. You ONLY use numbers provided in the [DATA] block below. 
   Never calculate, estimate, or infer amounts yourself.

   say exactly: "I don't have enough data to answer that yet."

   one short paragraph max for complex ones).

8. Hallucination Guardrails
This section is non-negotiable for a financial product.
8.1 The Problem
LLMs will occasionally invent numbers, especially when:
The context is ambiguous
The user asks a compound question
The model tries to be helpful by "filling in" missing data
In a financial app, an invented number (e.g., "you have about 500,000 RWF in savings") could cause real harm.
8.2 Output Validation Layer
Every LLM response passes through a validator before being shown to the user:
class OutputValidator:
    
    def validate(self, llm_response: str, engine_data: dict) -> ValidationResult:
        
        # Step 1: Extract all numbers from LLM response
        numbers_in_response = self.extract_numbers(llm_response)
        
        # Step 2: Build set of "allowed numbers"
        allowed = self.build_allowed_set(engine_data)
        # Includes: exact values, ±5% tolerance range, 
        # derived values (percentages of amounts in data)
        
        # Step 3: Check each extracted number
        for number in numbers_in_response:
            if not self.is_allowed(number, allowed):
                # Number in response not derivable from engine data
                return ValidationResult(
                    passed=False,
                    issue="hallucinated_number",
                    offending_value=number
                )
        
        # Step 4: Check for absolute claim keywords without data support
        absolute_claims = ["definitely", "certainly", "guaranteed", 
                          "exactly", "precisely", "will be"]
        if engine_data.get("confidence") == "low":
            for claim in absolute_claims:
                if claim in llm_response.lower():
                    return ValidationResult(passed=False, issue="overconfident_claim")
        
        return ValidationResult(passed=True)
    
    def on_validation_failure(self, failure: ValidationResult) -> str:
        # Retry once with stricter prompt
        # If second attempt also fails: return safe fallback
        return self.safe_fallback_response(failure)
8.3 Safe Fallback Responses
If validation fails twice, show a rule-based response instead:
Engine result: "affordable: true, months_needed: 4"
Safe fallback:
"Based on your current savings rate, you're on track to 
 afford this in approximately 4 months. Would you like 
 to see a detailed breakdown?"
This is less natural than an LLM response, but always correct.
8.4 Confidence Disclosure
When the engine returns confidence: "low", the AI must include a qualifier:
Engine: confidence = "low" (due to income variance)
LLM response must include one of:
  "Based on your recent data, which has been variable..."
  "This is an estimate — your income has fluctuated recently..."
  "I'd recommend revisiting this closer to December as your 
   income pattern becomes clearer..."
Validate that one of these qualifiers is present. If not, inject one.
9. Database Schema
9.1 Core Tables
-- Users
CREATE TABLE users (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    email       VARCHAR(255) UNIQUE NOT NULL,
    phone_hash  VARCHAR(64),              -- hashed MTN/Airtel number
    created_at  TIMESTAMPTZ DEFAULT NOW(),
    ai_mode     VARCHAR(20) DEFAULT 'smart' CHECK (ai_mode IN ('smart', 'private')),
    currency    VARCHAR(3) DEFAULT 'RWF'
);
-- Wallets (supports multiple: bank, MoMo, cash)
CREATE TABLE wallets (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         UUID REFERENCES users(id) ON DELETE CASCADE,
    name            VARCHAR(100) NOT NULL,     -- "MTN MoMo", "Equity Bank"
    type            VARCHAR(20) CHECK (type IN ('mobile_money', 'bank', 'cash', 'savings')),
    balance_rwf     BIGINT NOT NULL DEFAULT 0, -- in RWF, no decimals
    is_primary      BOOLEAN DEFAULT FALSE,
    created_at      TIMESTAMPTZ DEFAULT NOW()
);
-- Transactions
CREATE TABLE transactions (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         UUID REFERENCES users(id) ON DELETE CASCADE,
    wallet_id       UUID REFERENCES wallets(id),
    amount_rwf      BIGINT NOT NULL,           -- positive = credit, negative = debit
    type            VARCHAR(20) CHECK (type IN ('income', 'expense', 'transfer')),
    category_id     UUID REFERENCES categories(id),
    category_source VARCHAR(20) CHECK (category_source IN ('user', 'merchant_table', 'keyword', 'llm', 'uncategorized')),
    merchant_name   VARCHAR(255),              -- stripped before LLM use
    description     TEXT,
    transaction_at  TIMESTAMPTZ NOT NULL,
    is_recurring    BOOLEAN DEFAULT FALSE,
    is_flagged      BOOLEAN DEFAULT FALSE,
    flag_reason     VARCHAR(50),
    source          VARCHAR(20) CHECK (source IN ('manual', 'sms_import', 'api_import')),
    created_at      TIMESTAMPTZ DEFAULT NOW()
);
-- Categories
CREATE TABLE categories (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name        VARCHAR(100) NOT NULL,
    slug        VARCHAR(50) UNIQUE NOT NULL,   -- 'food', 'transport', etc.
    icon        VARCHAR(50),
    is_system   BOOLEAN DEFAULT TRUE,          -- false = user-created
    user_id     UUID REFERENCES users(id)      -- null for system categories
);
-- Budgets
CREATE TABLE budgets (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         UUID REFERENCES users(id) ON DELETE CASCADE,
    category_id     UUID REFERENCES categories(id),
    amount_rwf      BIGINT NOT NULL,
    period          VARCHAR(10) CHECK (period IN ('monthly', 'weekly')),
    active_from     DATE NOT NULL,
    active_to       DATE,                      -- null = ongoing
    created_at      TIMESTAMPTZ DEFAULT NOW()
);
-- Goals
CREATE TABLE goals (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         UUID REFERENCES users(id) ON DELETE CASCADE,
    name            VARCHAR(255) NOT NULL,
    target_rwf      BIGINT NOT NULL,
    current_rwf     BIGINT NOT NULL DEFAULT 0,
    target_date     DATE,
    status          VARCHAR(20) DEFAULT 'active' CHECK (status IN ('active', 'completed', 'paused', 'cancelled')),
    created_at      TIMESTAMPTZ DEFAULT NOW()
);
-- Financial Health Score History
CREATE TABLE health_scores (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         UUID REFERENCES users(id) ON DELETE CASCADE,
    score           SMALLINT CHECK (score BETWEEN 0 AND 100),
    savings_rate    SMALLINT,
    emergency_fund  SMALLINT,
    budget_adherence SMALLINT,
    income_stability SMALLINT,
    debt_servicing  SMALLINT,
    computed_at     TIMESTAMPTZ DEFAULT NOW()
);
-- AI Conversation Sessions
CREATE TABLE ai_sessions (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         UUID REFERENCES users(id) ON DELETE CASCADE,
    started_at      TIMESTAMPTZ DEFAULT NOW(),
    last_active_at  TIMESTAMPTZ DEFAULT NOW()
);
-- AI Messages (conversation history)
CREATE TABLE ai_messages (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    session_id      UUID REFERENCES ai_sessions(id) ON DELETE CASCADE,
    role            VARCHAR(10) CHECK (role IN ('user', 'assistant')),
    content         TEXT NOT NULL,
    intent          VARCHAR(50),               -- classified intent
    engine_data     JSONB,                     -- what engine returned for this turn
    validation_passed BOOLEAN,
    created_at      TIMESTAMPTZ DEFAULT NOW()
);

11. Phased Roadmap
Phase 1 — Foundation (Months 1–3)
Goal: Working product with no AI.
User authentication (JWT)
Wallet management
Manual transaction entry
Category management (system + custom)
Budget creation and tracking
Basic dashboard (balances, recent transactions, budget gauges)
MoMo SMS import (parsing only, user confirms each)
Definition of done: A user can track their spending and see if they're over budget. Nothing else.
Phase 2 — Intelligence Without AI (Months 4–5)
Goal: Useful insights from data, no LLM yet.
Financial Health Score (computed nightly)
Spending analysis (category trends, z-score alerts)
Affordability calculator (form-based, not chat)
Rule-based budget recommendations
Goals with progress tracking
Definition of done: The Financial Engine exists and is fully tested. Scores and analysis are shown in the UI as data cards, not chat.
Phase 3 — LLM Integration (Months 6–8)
Goal: Natural language interface over existing engine.
AI Orchestrator service (Python FastAPI)
Intent classifier
Context builder
LLM integration (OpenAI GPT-4o-mini)
Output validator
Two-mode privacy setting (Smart / Private)
Rate limiting (20 AI queries/user/day)
Chat UI in mobile app
Definition of done: A user can ask "Can I afford a laptop?" and get a correct, validated answer backed by their real data.
Phase 4 — ML Models (Month 9+)
Trigger: ≥ 500 users with ≥ 6 months of data.
Expense forecasting model (LightGBM)
Smarter transaction categorization
Personalized budget recommendations
Anomaly detection (IQR-based → model-based)
A/B test: ML forecast vs. rolling average baseline
Only ship ML if it beats the baseline on held-out test data
